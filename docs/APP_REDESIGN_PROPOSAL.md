## 1. 设计目标与原则

**目标**：把 App 从"单组查询工具"升级为贴着论文四步工作流的**实验决策台**——服务第③步（ML 打分 + 条件规划），并向第④步（RAG 迭代）提供标准化数据出口。

**原则**：

1. **场景优先**：真实场景是"化学家圈定 4–5 组单体（17 个组合），要排序、要方案、要记录"，不是匿名 SMILES 查询
2. **诚实输出**：打分 + 不确定度 + OOD 三级标记，界面统一用「成膜打分（倾向性）」口径，不称"概率"
3. **数据沉淀**：每一次查询、每一组收藏、每一份实验记录都是结构化资产，供 RAG 消费、供未来重训
4. **不重造闭环**：RAG 迭代的主体在 minimax 项目（shiyandiedai），本 App 只做标准化数据的生产者和展示端

## 3. 各页详细设计

### ① 查询打分（单组深度分析）

**输入区（三种方式并存）**：
- **SMILES 直输**：醛 SMILES + 胺 SMILES（现有）
- **CAS 号查询**：输入 CAS → 解析为 SMILES → 自动填入
  - 解析链路：本地缓存（`data/cas_cache.json`）→ PubChem PUG-REST API（`compound/name/{cas}/property/CanonicalSMILES`）→ RDKit 校验
  - 内置单体（TAPT/TFPT/TAPB/A1–A7/B1–B6）的 CAS 从方案文档预置，离线可用
  - 解析失败：明确报错（"未找到该 CAS / 需联网"），不静默猜
- **内置单体库点选**：下拉/卡片选择已知单体（带结构缩略图），免输入

**输出区**（现有能力全部保留 + 新增）：
- 成膜打分 ± 不确定度（bagging），路由臂与原因，GNN 对照分
- OOD 标记：⛔ 不适用（非标准官能团）/ ⚠️ 外推警告（双未见、超分布包络）
- SHAP 中文打分理由（分组贡献 + Top± 特征）
- 结构图三件套：醛 / 胺 / 缩合产物骨架
- **相似成膜案例**：训练文献中醛/胺各自最接近的成功配对（含文献出处）——数据现成，新做
- 「☆ 收藏这组单体」按钮（→ 页③）和「生成实验方案」按钮（→ 页③方案卡）
- Word 报告下载（现有，同步新字段）

### ② 批量排序

- 输入：内置库多选 / 粘贴 SMILES 列表 / 上传 CSV
- 输出**排序表**：打分 ± std、路由臂、OOD 状态、Top 理由一句话、可点击展开单行详情
- 按打分 / 不确定度 / OOD 过滤排序；一键导出 Word / CSV
- 直接服务"17 个组合排优先级"的真实场景

### ③ 收藏夹（核心新增）

**收藏条目结构**（`data/favorites/<id>.json` + 附件目录）：

```json
{
  "id": "fav_20260721_001",
  "aldehyde": {"smiles": "...", "cas": "...", "name": "A2"},
  "amine":    {"smiles": "...", "cas": "...", "name": "TAPT"},
  "created_at": "...", "notes": "用户备注",
  "latest_prediction": {"score": 0.699, "std": 0.04, "arm": "tree_v4",
                        "ood": "none", "date": "..."},
  "references": [
    {"title": "...", "doi": "...", "source": "auto-matched|user-added",
     "path_or_url": "...", "note": "支撑哪条决策"}
  ],
  "experiment_record_ids": ["rec_001"]
}
```

**功能**：
- 收藏列表卡片墙：结构图 + 打分徽章 + OOD 状态
- **文献附件两种方式**：
  - **自动匹配**：训练语料来自 ~955 篇结构化文献（`旧项目/data/structured/`），按醛/胺 paper_id 反查"哪些文献报道过这对/这个单体"，自动挂为参考（这是别家做不出的功能，数据就在我们手里）
  - **手动添加**：用户填标题/DOI/备注，或上传 PDF 到附件目录
- 收藏详情页：历史打分变化、关联实验记录列表、「生成/更新实验方案」

### ④ 实验记录

- **上传表单**：选收藏条目 → 填实际条件（溶剂/调制剂/催化剂/温度/时间/加料顺序）→ 结果（成膜/部分/失败 + 机械强度描述）→ 备注 + 附件（照片/docx）
- 存为标准化 JSON（`data/experiment_records/<id>.json`），**独立 record_id**（继承旧项目教训：合成样本必须有独立 id 防 CV 泄漏）
- 记录列表 + 与当初预测的偏差展示（"预测 0.65 / 实际：失败（乙酸过量）"）
- 每条记录 = 与用户分布完全一致的训练样本，攒够即触发重训评估

### ⑤ 方案迭代（RAG 对接页）

**边界**：迭代智能在 minimax RAG（未完成），本页是**数据出口 + 建议回显**。

三阶段对接：

| 阶段 | 方式 | 内容 |
|---|---|---|
| 1（现在） | **文件对接** | App 把预测、收藏、实验记录按约定 schema 写入 `data/rag_export/`；RAG 直接读取，零耦合 |
| 2 | **API 对接** | Gradio 自带 API 暴露 `predict(smiles)` / `get_records()`，RAG 远程调用 |
| 3 | **建议回显** | RAG 产出的迭代建议 JSON（如"G2 建议把苯胺降到 X，依据文献 Y"）落回 `data/rag_export/suggestions/`，本页展示并与收藏条目关联 |

**本页展示**：实验记录时间线 → 每轮的预测 vs 实际 → RAG 建议列表（阶段 3 到位后）。

## 4. 实验方案设计（嵌入页①/③）

按侯老师法模板生成**可核对方案卡**：

- 条件参数：溶剂（甲苯/氯仿，G5 类用 BTF/二氧六环）、苯胺调制剂用量、6M 乙酸、120 °C、2–5 天、35 mL Pyrex 管
- **加料顺序与操作要点**
- **防错清单**（来自实验 ABCDEF 的真实失败教训）：乙酸浓度核对（6M≠18M）、溶解完全性检查、苯胺量按方案、密封与排气
- 单体特异提示：F/CF3 单体溶解性、酰肼（H 组）显示"腙键体系，模型不适用但方案可给"
- 导出 Word（复用现有 exporter）

## 5. 视觉美化方案

继续用 Gradio 6（不引入新框架，成本最低），但彻底改版：

- **主题**：`gr.themes.Soft` 定制主色（建议深青/石墨色系，学术感），统一中文字体栈
- **布局**：顶部标题栏（项目名 + 模型版本徽章）→ 标签页；卡片式容器（`gr.Group`），主输出两栏（左分数右结构图）
- **色彩语义**：⛔ 红（不适用）/ ⚠️ 琥珀（外推）/ ✓ 绿（池内可信）；打分用大字号 + 颜色梯度
- **表格**：`gr.Dataframe` 排序表，行内嵌结构缩略图（页②）
- 细节：加载动画、错误提示人话化、空状态引导文案

## 6. 数据契约摘要（RAG 对接基础）

- `prediction`：{smiles 对， score, std, arm, ood, shap_top, model_version, timestamp}
- `experiment_record`：{record_id, favorite_id, conditions{…}, outcome, strength, notes, attachments, operator, date}
- `suggestion`（RAG 产出）：{suggestion_id, favorite_id, type: condition_adjust|new_candidate, payload, evidence_refs, created_at}
- 全部 JSON，版本号字段 `schema_version` 起步即带

另：每次查询写 `data/prediction_log.jsonl`（支持 D23 路由复盘点、使用统计）。

## 8. 风险与开放问题

1. **CAS→SMILES 依赖 PubChem 在线服务**：离线时仅内置库可用（可接受，需文案明示）
2. **文献自动匹配质量**：paper_id 反查得到的是"报道过该单体的文献"，与"支撑该组合"不完全等同，UI 上措辞为"相关文献"
3. **RAG schema 需要与 minimax 侧对齐一次**：建议 P3 开工前双方（你 + 我）过一遍第 6 节契约
4. 收藏/记录数据量增长后 JSON 文件是否换 SQLite：起步用 JSON（可读、git 友好），超 ~500 条再评估
