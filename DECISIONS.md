## D01 — 任务形态：推荐/打分排序

**决定**：项目核心任务从"二分类（能否成膜）"改为"推荐/打分排序"。

**为什么**：
- 用户问题 1：样本严重失衡（正样本远多于负样本），本质是发表偏差导致负样本 MNAR（非随机缺失）。
- 强行二分类 → 必须造负样本 → 负样本不可信 → 标签噪声放大。
- 推荐任务用"打分+排序+多样性约束"替代二分类，天然规避负样本缺失困境，更贴合"实验指导工具"的最终交付。

**考虑过的替代方案**：
- 条件分类（预测"某组合+给定条件下成膜概率"）：保留作为模型内部机制，但对外形态仍是打分排序。
- 纯二分类：被否决，负样本问题是死结。

**何时复盘**：阶段 4 完成首次建模后，评估排序指标（NDCG/Hit@K）是否能有效区分好坏组合。

## D02 — 主模型：树模型+描述符（GNN 并行）

**决定**：以 **XGBoost/LightGBM + 化学描述符**为**主可解释模型**；**GNN 作为并行准确率模型**，两者不是替代关系。项目成功标准为：
- (a) 树模型在留一单体 CV 上超越 GNN 基线（PR-AUC 0.784）；或
- (b) 树模型接近 GNN 但能给出不可替代的化学解释（SHAP 揭示规则）。

**为什么**：
- 用户问题 2、8：旧项目走 GNN 路线，泛化仍差，RF 又"像相似度排序"。
- 核心判断：**泛化差和相似度排序不是模型架构问题，是描述符尺度问题 + 样本量问题**。
  - GNN 在 ~6000 样本上学到的仍是表面相似性 → 体现为筛选阶段泛化差。
  - RF "像相似度排序"是因为描述符都是绝对量（分子量/原子数），小单体特征空间天然更近 → 退化成最近邻。
- **但旧项目 GNN 已达 PR-AUC 0.784，这是强基线**。在没证明树模型能超越之前，不应把 GNN 降级为"陪跑"，而应保留为并行模型。
- 树模型在条件特征利用和可解释性上有天然优势，GNN 在端到端学习已验证的结构模式上有优势。两者可以互补：树模型用于解释+条件推理，GNN 用于最终打分或验证。

**考虑过的替代方案**：
- 树模型为主，GNN 仅作对比基线：被否决，风险过高（若树模型未达 0.784，项目无退路）。
- GNN 为主，树模型辅助：被否决，无法解决条件特征未利用和可解释性需求。

**何时复盘**：阶段 4 首次建模，对比树模型 vs GNN 在留一单体交叉验证（leave-one-monomer-out）下的泛化能力和解释性。

## D03 — 反应条件：首次作为特征纳入

**决定**：把反应条件（温度/溶剂/催化剂/合成路线/界面类型）作为特征纳入模型。

**为什么**：
- 用户问题 3：粉末≠不能成膜，成膜与否对反应条件极敏感。丢掉条件等于丢掉一半信息。
- **关键审阅发现**：旧项目 `v5_train_stage1.csv` 已有完整反应条件字段（16 列），但 **GNN 从未使用过它们**（只用结构+规则向量）。旧项目规划过 Stage 2 条件预测（FiLM 调制）但未实现。
- 因此"重新纳入条件"实际上是**首次真正使用条件特征**——这是新项目相对旧项目最大的增量突破点。

**考虑过的替代方案**：
- 不纳入条件（纯结构）：被否决，用户明确选择纳入，且化学上必要。
- 纳入但作为辅助任务（旧项目 ConditionHead 方案）：被否决，多任务在小数据上梯度冲突（v3 崩溃教训）。

**实现要点**：
- 条件字段编码：温度（数值归一化）、溶剂（分类 one-hot 或嵌入）、催化剂（分类）、合成路线（分类）、界面类型（分类）。
- 排序时条件作为输入的一部分：输入"组合+条件"，输出成膜概率。

**何时复盘**：阶段 3 特征工程完成后，消融实验对比"有/无条件特征"对模型性能的影响。

## D04 — 标签体系：继承四分类连续

**决定**：继承旧项目四分类连续标签体系。

**标签映射**：
| 标签值 | 含义 | 旧项目样本数 |
|---|---|---|
| 1.0 | 文献确认连续成膜 | 429 |
| 0.8 | 粉末合成成功但非连续膜（弱正） | 291 |
| 0.7 | 增广正样本（衍生自文献正，quality_weight=0.3） | 1,308 |
| 0.0 | 负样本（化学规则生成+文献内负样本） | 4,173 |

**为什么**：
- 用户问题 3：粉末不应直接当负样本。旧项目已用 0.8 弱正处理，比二元更合理。
- 连续标签让模型学到"程度"而非"是否"，与推荐任务（打分排序）天然契合。

**何时复盘**：阶段 1 数据审计时，重新评估 0.8 粉末样本的质量和数量，可能需要细分（成膜条件不当 vs 结构性失败）。

## D05 — 规则向量：作为特征（非硬过滤）

**决定**：继承旧项目 34 维规则向量作为树模型特征，而非推理后硬过滤。

**为什么**：
- 用户问题 5："把规则嵌进描述符对不对"。答案：**取决于规则来源**。
  - ✅ 领域规则（化学/文献依据，如"Schiff base 需胺醛官能团匹配"）当特征 = 注入可靠归纳偏置。
  - ❌ 拟合规则（从训练集拟合的）当特征 = 数据泄露/过拟合。
- 旧项目 34 维规则中：23 维是化学领域规则（苯环必需/官能团对称等），11 维是 F/CF3 描述符，**都是领域规则，当特征合理**。
- 旧项目已验证：规则注入 FilmHead（非硬过滤）让模型自行学习规则权重，比硬过滤效果好。

**何时复盘**：阶段 4，对比"有/无规则特征"的模型表现，以及 SHAP 能否独立学到这些规则模式（如果 SHAP 自动学到，说明规则可去掉；学不到，说明规则特征必要）。

## D08 — GNN：作为并行准确率模型（非仅对比基线）

**决定**：旧项目 GNN v5（PR-AUC 0.784）作为新项目的**并行准确率模型**，与树模型共同发展，不是等待被超越的基线。

**为什么**：
- 旧项目 GNN 不是"完全没做好"，PR-AUC 0.784 是强基线。
- 新项目同时追求：树模型利用条件特征 + 提供可解释性；GNN 保持准确率上限。
- 用户感受到的"泛化差/相似度排序"主要在**筛选阶段**（笛卡尔积+多样性过滤），需区分训练性能 vs 筛选性能。
- 最终交付可以是两者结合：树模型做初筛/解释，GNN 做精排/验证，或ensemble。

**成功标准**（待阶段 4 细化）：
- 树模型在留一单体 CV 上 PR-AUC 接近或超越 GNN；或
- 两者 ensemble 后优于任一单一模型；或
- 树模型提供 GNN 无法给出的化学可解释洞察。

## D10 — 交付形态：Jupyter notebook 优先

**决定**：第一阶段实验指导工具以 Jupyter notebook 形式交付，而非 CLI 或 Web 界面。

**为什么**：
- Notebook 最适合探索性分析和可视化（SHAP 图、分子结构图、相似案例）
- 开发速度快，便于调试
- 可后续封装为 CLI 脚本或 Web 界面

**交付标准**：
- 输入：候选组合 DataFrame（含醛/胺 SMILES + 条件）
- 输出：打分排序 DataFrame + SHAP 解释 + 相似案例
- 可视化：分子结构、特征重要性、Top-K 推荐卡片

**何时复盘**：阶段 4 建模完成后，根据使用习惯决定是否封装为 CLI/Web。

## D18 — 启用 joblib 文件级构象缓存

**决定**：为 3D/二聚体描述符计算启用文件级缓存，默认路径 `data/interim/3d_cache/`。

**为什么**：
- 全量 6201 对计算 3D+二聚体约需 42 分钟；同一单体在多个配对中复用，单体缓存可显著减少重复计算。
- 二次读取缓存后，10 行特征化从数秒降到 0.02 秒，全量特征化可从分钟级降到秒级。
- joblib dict 格式轻量，写入时先写 `.tmp` 再重命名，可防止中断损坏。

**缓存策略**：
- 单体 3D：key = SMILES 字符串
- 二聚体 3D：key = (ald_smiles, amine_smiles) 元组
- 缓存文件不进入 git（路径在 `data/interim/` 下，已被 gitignore 覆盖）

**何时复盘**：500 对验证后，若缓存文件过大或读写成为瓶颈，可改为按分子哈希分片的 `.npy`/`.parquet`。

## D19 — tree_v3 采用单体 3D 描述符，不启用二聚体 3D

**决定**：将 `tree_v3_monomer3d_full` 选为新主模型 `tree_v3`，即启用单体 3D 描述符，但不启用二聚体 3D 描述符。

**为什么**：
- 全量验证显示 `monomer3d_full` PR-AUC 0.7785、MAE 0.2344，均优于 `dimer3d_full`（PR-AUC 0.7716、MAE 0.2395）。
- `dimer3d_full` 的 NDCG@10 更高（0.9216 vs 0.9055），但 PR-AUC 和 MAE 略差，说明二聚体几何信息部分已被单体 3D + Hadamard 交互覆盖，存在冗余。
- 项目核心目标之一是输出准确成膜概率，PR-AUC 和 MAE 是更直接的指标。

**何时复盘**：
- 若后续发现排序/Top-K 场景更重要，可重新评估二聚体 3D 或仅使用二聚体 3D 做排序模型。
- 可进一步做仅二聚体 3D（无单体 3D）的消融，确认其独立贡献。

## D20 — App/predictor 默认模型切换为 tree_v3，特征开关从 pkl 自动恢复

**决定**：
1. `FilmPredictor` 默认树模型从 `tree_baseline.pkl` 切换为 `models/tree_v3.pkl`。
2. `TreeFilmPredictor.load()` 从 pkl 内的 `metrics` 恢复特征开关（`use_rules` / `reduced_rules` / `use_interaction` / `use_3d` / `use_dimer` / `n_confs`），预测时按训练配置生成特征；旧 pkl 无 `metrics` 则回退默认特征。
3. 预测特征矩阵用 `reindex(columns=feature_cols)` 对齐，3D 计算失败的样本补 0 而不是报错。

**为什么**：
- tree_v3 有 142 维特征（含单体 3D），若预测时仍按默认开关生成特征，会因缺列直接报 KeyError；开关存进 pkl 后模型自描述，切换模型不需要改代码。
- App 用户可输入任意 SMILES，3D 构象生成可能失败，补 0 比中断更稳健。
- `TreeFilmPredictor` 的默认路径保持 `tree_baseline.pkl` 不变：`tests/test_app_pipeline.py` 的快速训练测试会保存到默认路径，避免测试覆盖主模型。

**何时复盘**：若后续切换主模型（如 tree_v4），只需改 `src/predictor/__init__.py` 中的 `DEFAULT_TREE_MODEL`。
